# Alt-Opinion Directive: infra-base Spec Validation

## intent:

Проведи свежий, критический разбор спецификации infra-base. Это автономная AI-разработка — оператор не будет отвечать на промежуточные вопросы, все решения принимаются агентом.

**Что оценить:**

1. **Tool Stack:** Является ли стек оптимальным для local-first PWA на Svelte 5? Есть ли неучтённые инструменты, которые критически необходимы (например, для офлайн-синхронизации, service worker, IndexedDB)? Не пропущена ли категория, которая станет блокером?

2. **Decision Log:** Достаточно ли обоснованы решения D-001..D-013? Есть ли противоречия между решениями? Например:
   - D-002 (npm) vs отсутствие pnpm в rejected alternatives — оправдано ли?
   - D-005 (Biome) — достаточен ли Biome для Svelte-файлов (линтер Svelte не поддерживается)?
   - D-009 (Svelte 5 standalone vs SvelteKit) — правильно ли выбрано?
   - D-011 (Firebase) — vendor lock-in приемлем?

3. **Uncomfortable Questions:** Какие неудобные вопросы НЕ были заданы?
   - Как обеспечивается офлайн-работа (IndexedDB, Service Worker)?
   - Где хранятся секреты (Firebase config)?
   - Как решается конфликт синхронизации между устройствами?
   - Что если npm registry недоступен?
   - Какая стратегия версионирования пакетов (^ vs exact)?
   - Достаточен ли один `check-command` для всех проверок или нужен CI-grade?

4. **Effective Rules:** Все ли правила из knowledge.xml корректно сопоставлены с инструментами? Пропущены ли какие-то правила? Правильно ли обработано отсутствие Biome-правила?

5. **Scope Dependencies:** Правильно ли определена структура: infra-base → domain-lib → web? Не должен ли domain-lib также зависеть от чего-то ещё?

6. **Bootstrap Requirements:** Все ли внешние зависимости учтены? Что насчёт `@sveltejs/vite-plugin-svelte`, `@playwright/test`, `firebase-tools`?

7. **Dev Workflow Example:** Реалистичен ли flow? Покрывает ли он все сценарии (первый запуск, daily dev, CI)?

8. **Связанность с Portal (specs/README.md):** Соответствует ли Scope Graph спецификации? Корректны ли описания?

**Формат ответа:**
- Критические проблемы (блокеры)
- Существенные замечания (должны быть исправлены)
- Рекомендации (nice-to-have)
- Пропущенные uncomfortable questions с предлагаемыми ответами
- Итоговый вердикт: готово к scaffold / требует доработок
